Foundation
Cloud Computing Basics
Understand what the cloud is, why it exists, and why AWS dominates it.
☁️

What is Cloud Computing?

Cloud computing is the delivery of computing services — servers, storage, databases, networking, software — over the internet ("the cloud") so you don't have to own or maintain physical hardware.

💡 ANALOGY

Think of cloud computing like electricity. You don't build your own power plant — you just plug in and pay for what you use. AWS is the power company for your apps and data.

⚡
Benefits of Cloud Computing
▶
  • Trade Capital Expense for Variable Expense — Instead of buying servers upfront, pay only for what you consume.
  • Massive Economies of Scale — AWS serves millions of customers, so they can offer lower prices than you could achieve alone.
  • Elasticity — Scale up during peak traffic (Black Friday), scale down during quiet periods. Pay only for current use.
  • Speed & Agility — Provision new resources in minutes, not weeks.
  • Global Reach — Deploy your application worldwide in a few clicks.
  • No Data Center Management — AWS handles physical security, hardware maintenance, and cooling.
📦
Cloud Service Models: IaaS, PaaS, SaaS
▶

🏗️ IaaS

Infrastructure as a Service

You manage OS, runtime, data. AWS provides hardware.
Example: EC2 (virtual machines)

🚀 PaaS

Platform as a Service

AWS manages OS + runtime. You manage your code and data.
Example: Elastic Beanstalk, RDS

💻 SaaS

Software as a Service

You just use the software. Everything managed by provider.
Example: Gmail, Salesforce

💡
Most AWS services are IaaS or PaaS. As an AWS Technician, you'll primarily work with IaaS services like EC2 and PaaS services like RDS.
🌐
Deployment Models: Public, Private, Hybrid
▶

Public Cloud

Resources owned and operated by a third-party cloud provider (AWS). Everything runs on the provider's hardware. Best for most businesses.

Private Cloud

Cloud resources used exclusively by one organization. Can be on-premises or hosted by a third party. More control, but higher cost.

Hybrid Cloud

A mix of on-premises + public cloud. Data/apps can move between the two. Common in regulated industries (banking, healthcare).

🏆

02
Infrastructure
AWS Global Infrastructure
Regions, Availability Zones, and Edge Locations — the backbone of AWS.
🌍

AWS operates a massive global network of data centers organized into a hierarchy: Regions → Availability Zones → Edge Locations.

// AWS GLOBAL INFRASTRUCTURE HIERARCHY
🌍
Regions
▶

A Region is a physical geographic location in the world where AWS has clustered data centers. AWS currently has 33+ Regions globally (e.g., us-east-1, eu-west-1, ap-southeast-1).

How to choose a Region?

  • Compliance & Data Residency — Government regulations may require data to stay in-country.
  • Latency — Choose the region closest to your end users for fastest response.
  • Service Availability — New AWS services roll out to us-east-1 first.
  • Pricing — Costs vary by region (us-east-1 is typically cheapest).
⚠️
Important: Resources in one Region are completely isolated from another Region by default. S3 in us-east-1 is separate from S3 in eu-west-1.
🏢
Availability Zones (AZs)
▶

Each Region has multiple Availability Zones (AZs) — usually 3 or more. Each AZ is one or more discrete data centers with redundant power, networking, and connectivity.

💡 ANALOGY

Think of a Region as a city (New York), and AZs as different neighborhoods in that city. If one neighborhood floods, the others are fine. Your app runs in multiple neighborhoods simultaneously.

  • AZs are physically separated by many kilometers to prevent simultaneous failure.
  • They are connected by high-speed, low-latency private fiber links.
  • Deploying to multiple AZs makes your app highly available.
📡
Edge Locations & CloudFront
▶

Edge Locations are mini data centers distributed globally (400+ locations) used to deliver content to end users with low latency. They are used by Amazon CloudFront (CDN) and Route 53 (DNS).

💡 ANALOGY

If AWS Regions are warehouses, Edge Locations are the local convenience stores. Instead of ordering from a central warehouse (farther away), you pick up from the store around the corner — faster delivery!

Security
IAM — Identity & Access Management
Control who can do what in your AWS account. The foundation of all AWS security.
🔐

IAM is a free AWS service that enables you to manage access to AWS resources. It's the first thing you configure after creating an AWS account.

🚨
Critical Rule: Never use the root account for daily tasks. Create IAM users with only the permissions they need (Principle of Least Privilege).

IAM Users

An IAM User represents a person or service that interacts with AWS. Each user has:

  • A unique username within the account
  • A password (for Console access) or Access Keys (for programmatic/CLI access)
  • Permissions attached directly or through groups
💡 ANALOGY

An IAM User is like an employee badge. Each employee gets their own badge (identity) with specific access doors (permissions) they can open.

IAM Groups

A Group is a collection of IAM users. Attach policies to groups to manage permissions at scale.

Example Groups

Developers, Admins, Finance, ReadOnly, DBAdmins

Key Points

A user can belong to multiple groups. Groups cannot be nested inside other groups.

IAM Roles

A Role is an identity with specific permissions that can be assumed by AWS services, users, or external applications — not tied to a specific person.

  • EC2 Role — Lets an EC2 instance access S3 without embedding credentials
  • Lambda Role — Allows a Lambda function to write to DynamoDB
  • Cross-Account Role — Let users from Account A access resources in Account B
💡 ANALOGY

A Role is like a costume. Any actor (service) can wear it temporarily to gain specific abilities, then take it off. The costume defines what you can do, not who you are permanently.

IAM Policies

A Policy is a JSON document defining what actions are allowed or denied on which resources.

IAM Policy — Allow S3 Read Access 
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",     // Allow or Deny
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}
💡
Policy evaluation: Deny wins. If any policy denies an action, it's denied even if another allows it. Default is deny — you must explicitly grant access.
🔑
MFA — Multi-Factor Authentication
▶

MFA adds a second layer of security on top of username and password. Even if your password is stolen, the attacker can't log in without your MFA device.

  • Virtual MFA — Google Authenticator, Authy (time-based OTP)
  • Hardware MFA — Physical device (YubiKey)
  • Best Practice: Enable MFA on the root account AND all admin IAM users
🪪

Compute
EC2 — Elastic Compute Cloud
Virtual servers in the cloud. The most fundamental AWS compute service.
🖥️

EC2 provides resizable virtual servers (instances) in the cloud. You choose the OS, size, and configuration — and start running applications in minutes.

💡 ANALOGY

EC2 is like renting a computer in a data center. You tell AWS: "I want a computer with 8 CPUs, 32GB RAM, running Linux" — and it's ready in 60 seconds. You pay by the hour. Done? Turn it off.

📊

Instance types define the hardware (CPU, memory, network). Named like t3.micro (family + generation + size).

💰
EC2 Pricing Models
▶

🕐 On-Demand

Pay per second/hour

No commitment. Launch and terminate anytime. Most flexible but most expensive. Best for unpredictable workloads.

📝 Reserved Instances

1 or 3 year commitment

Up to 72% discount vs On-Demand. Best for steady-state workloads (production databases, app servers).

⚡ Spot Instances

Up to 90% discount

Use spare AWS capacity. Can be interrupted with 2-min notice. Best for batch jobs, ML training, fault-tolerant apps.

🏠 Dedicated Hosts

Per host billing

Physical server dedicated to you. Required for compliance or software licensing. Most expensive.

🛡️
Security Groups
▶

Security Groups are virtual firewalls for EC2 instances. They control inbound and outbound traffic at the instance level.

  • All inbound traffic is denied by default — you must add rules to allow it.
  • All outbound traffic is allowed by default.
  • Rules are stateful — if you allow inbound, the response outbound is automatically allowed.
  • You can reference other Security Groups in rules (not just IP addresses).
Security Group Example — Web Server
// INBOUND RULES
Port 80   (HTTP)  — Source: 0.0.0.0/0     // Anyone
Port 443  (HTTPS) — Source: 0.0.0.0/0     // Anyone
Port 22   (SSH)   — Source: 10.0.0.5/32   // Only your IP

// OUTBOUND RULES
All traffic — Destination: 0.0.0.0/0        // All outbound allowed
💾
EBS — Elastic Block Store (EC2 Storage)
▶

EBS provides persistent block storage volumes for EC2 instances — like a hard drive you attach to your virtual machine.

gp3 / gp2 (SSD)

General purpose. Best for most workloads, boot volumes, databases.

io1 / io2 (SSD)

Provisioned IOPS. High-performance databases requiring consistent low-latency I/O.

st1 (HDD)

Throughput-optimized. Big data, data warehouses, log processing.

sc1 (HDD)

Cold HDD. Infrequently accessed data. Lowest cost EBS option.

✅
EBS volumes are tied to one AZ. They persist even after the EC2 instance is stopped/terminated (unless "Delete on Termination" is checked). You can create snapshots to S3 for backups.
📸
AMI — Amazon Machine Image
▶

An AMI is a template/snapshot used to launch an EC2 instance. It includes the OS, pre-installed software, and configuration.

  • AWS Managed AMIs — Amazon Linux 2, Ubuntu, Windows Server
  • Community AMIs — Shared by other AWS users
  • Custom AMIs — You create your own from a configured instance
  • AWS Marketplace AMIs — Commercial software (e.g., Cisco, Palo Alto)

Storage
S3 — Simple Storage Service
Unlimited, durable, scalable object storage. Store anything, anywhere.
🪣

S3 is one of AWS's most used services. It stores objects (files) in buckets (containers). Infinitely scalable, 99.999999999% (11 9s) durability.

💡 ANALOGY

S3 is like a giant Google Drive for developers. A bucket is like a folder. Objects are files. Unlike a hard drive, it never fills up, never breaks, and can be accessed by millions of users simultaneously.

🪣
Buckets & Objects
▶
  • Bucket names are globally unique across all AWS accounts.
  • Objects can be up to 5TB in size. Use Multipart Upload for files >100MB.
  • Objects have a key (path), value (content), and metadata.
  • Accessed via URL: https://my-bucket.s3.amazonaws.com/images/photo.jpg
💡
S3 is not a traditional file system — it has no folders. The "/" in "images/photo.jpg" is part of the key name, but the console shows it as a folder visually.
📁

▶

🔒
S3 Security & Access Control
▶

By default, S3 buckets are private. You control access with:

  • Bucket Policies — JSON-based policies at the bucket level (resource-based)
  • IAM Policies — Attached to IAM users/roles (identity-based)
  • ACLs — Legacy, per-object access control (mostly disabled now)
  • Block Public Access — Account-level switch to prevent any public access
  • S3 Encryption — SSE-S3, SSE-KMS, SSE-C, Client-Side Encryption
Bucket Policy — Public Read for Static Website
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": "*",         // Anyone (public)
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-website/*"
  }]
}
🌐
S3 Static Website Hosting
▶

S3 can host a static website (HTML, CSS, JS — no server-side code). Enable it in bucket settings and get a URL like:

http://my-bucket.s3-website-us-east-1.amazonaws.com
✅
Combine S3 static hosting with CloudFront CDN + Route 53 custom domain for a production-ready, serverless website at minimal cost.

Networking
VPC — Virtual Private Cloud
Your own isolated network inside AWS. The foundation of all networking.
🌐
💡 ANALOGY

A VPC is like buying a plot of land in the cloud. You decide how to divide it (subnets), build roads (route tables), install a fence (security groups, NACLs), and add a gate (Internet Gateway) for outside access.

// VPC ARCHITECTURE OVERVIEW
🔲
Subnets — Public vs Private
▶

🟦 Public Subnet

Has a route to the Internet Gateway. Resources here get public IP addresses and can be reached from the internet. Used for: Web servers, Load Balancers, Bastion hosts.

🟪 Private Subnet

No direct route to the internet. Resources are NOT publicly accessible. Used for: Databases, Application servers, internal microservices.

🛤️
Route Tables, IGW, NAT Gateway
▶
  • Internet Gateway (IGW) — Allows internet access for public subnets. One per VPC.
  • Route Table — Rules that determine where network traffic goes. Public subnet routes: 0.0.0.0/0 → IGW
  • NAT Gateway — Allows private subnet instances to access the internet (for updates) WITHOUT being exposed to inbound internet traffic. Lives in a public subnet.
  • VPC Peering — Connect two VPCs privately, even across accounts or regions.
🛡️
NACLs vs Security Groups
▶

LevelInstance levelSubnet level
Stateful?Yes — responses auto-allowedNo — must allow both directions
Allow/Deny rulesAllow onlyBoth Allow and Deny
Rule evaluationAll rules evaluatedRules evaluated in order (#)

07
Databases
RDS & DynamoDB — Databases on AWS
Managed relational and NoSQL databases. Store your application data.
🗄️

Amazon RDS — Relational Database Service

RDS is a managed service for relational databases. AWS handles provisioning, patching, backup, and replication — you just manage your data.

Supported Engines:

  • MySQL / MariaDB
  • PostgreSQL
  • Oracle Database
  • Microsoft SQL Server
  • Amazon Aurora (MySQL/PostgreSQL compatible)

Key Features:

  • Automated backups (up to 35 days retention)
  • Multi-AZ deployment for high availability
  • Read Replicas for read scaling (up to 15)
  • Automatic storage scaling
💡 ANALOGY

Running RDS vs setting up your own database is like hiring a full-time DBA (RDS) vs managing the database yourself. AWS handles all the maintenance work — you focus on queries.

Amazon DynamoDB — NoSQL Database

DynamoDB is a fully managed, serverless key-value and document NoSQL database designed for single-digit millisecond performance at any scale.

  • No servers to manage — AWS handles everything
  • Auto-scales from 0 to millions of requests/second
  • Stores items (rows) in tables with a Partition Key (required) and optional Sort Key
  • Flexible schema — each item can have different attributes
  • DynamoDB Streams — React to changes (like CDC) with Lambda
  • Global Tables — Multi-region, active-active replication

Use DynamoDB for:

Gaming leaderboards, shopping carts, session storage, IoT data, serverless apps

NOT good for:

Complex joins, transactions across many tables, ACID compliance at scale (use RDS)

Amazon Aurora

Aurora is AWS's own cloud-native database, compatible with MySQL and PostgreSQL but 5x faster than MySQL and 3x faster than PostgreSQL.

  • Storage auto-grows in 10GB increments up to 128TB
  • 6 copies of data across 3 AZs automatically
  • Up to 15 read replicas with minimal lag
  • Aurora Serverless — Auto-starts/stops and scales capacity based on demand
  • Best choice for new high-performance applications

Amazon ElastiCache

ElastiCache is a fully managed in-memory caching service. Use it to dramatically speed up database-heavy applications.

Redis

Rich data structures (lists, sets, sorted sets). Persistence. Pub/Sub. Best for session storage, leaderboards, real-time analytics.

Memcached

Simple key-value. Multi-threaded. Auto-discovery. Best for simple object caching when you need raw speed.

💡 ANALOGY

ElastiCache is your app's short-term memory. Instead of repeatedly asking the database (hard drive) for the same information, you remember it in RAM — instant recall.

Serverless Compute
Lambda & Serverless Architecture
Run code without managing servers. Pay only for execution time.
⚡
💡 ANALOGY

Lambda is like a vending machine for code execution. You put in a request (coin), your code runs (snack drops), and billing stops immediately. No need to own the vending machine — Amazon does.

⚡
AWS Lambda — How It Works
▶

Lambda runs your code in response to events without provisioning or managing servers.

  • Supports Python, Node.js, Java, Go, Ruby, .NET, and custom runtimes
  • Billed per request + duration (in ms). Free Tier: 1M requests/month free forever!
  • Max execution time: 15 minutes per invocation
  • Up to 10GB memory, 6 vCPUs (proportional to memory)
  • Each invocation is stateless — use DynamoDB/S3 for persistence
Lambda Function — Python Example
import json

def lambda_handler(event, context):
    # event = input data (from API Gateway, S3, etc.)
    name = event.get('name', 'World')
    
    return {
        'statusCode': 200,
        'body': json.dumps({
            'message': f'Hello, {name}!'
        })
    }
🎯

▶

Common Triggers

API Gateway, S3 events, DynamoDB Streams, SQS/SNS messages, CloudWatch Events (scheduled), ALB, EventBridge

Real Use Cases

Image resizing on upload, form processing, microservices APIs, scheduled reports, real-time file processing

Availability & Scale
Load Balancing & Auto Scaling
Automatically distribute traffic and scale capacity to meet demand.
⚖️
⚖️
Elastic Load Balancer (ELB) Types
▶

ELB distributes incoming traffic across multiple EC2 instances, containers, or Lambda functions for high availability.

ALB

Application Load Balancer

Layer 7 (HTTP/HTTPS). Routes by URL path, host headers, query strings. Best for web apps and microservices.

NLB

Network Load Balancer

Layer 4 (TCP/UDP). Handles millions of requests/second with ultra-low latency. Best for gaming, real-time apps.

GLB

Gateway Load Balancer

For deploying/scaling virtual network appliances (firewalls, IDS/IPS). Layer 3.

💡 ANALOGY

A Load Balancer is like a traffic cop at a busy intersection. Instead of all cars going to one lane (server), the cop directs them evenly across all available lanes — no lane gets overwhelmed.

📈
Auto Scaling Groups (ASG)
▶

Auto Scaling automatically adds or removes EC2 instances based on demand, ensuring optimal performance and cost.

Scaling Policies

  • Target Tracking — Keep CPU at 70%
  • Step Scaling — Add 2 instances when CPU > 80%
  • Scheduled — Scale up every Friday at 5 PM
  • Predictive — ML-based future scaling

ASG Configuration

  • Min — Never go below (e.g., 2 instances)
  • Max — Never exceed (e.g., 10 instances)
  • Desired — Current target count
  • Launch Template — AMI + instance type

CDN & DNS
CloudFront & Route 53
Global content delivery and intelligent DNS routing.
📡
⚡
Amazon CloudFront — CDN
▶

CloudFront is a Content Delivery Network (CDN) that delivers content from 400+ Edge Locations globally — bringing files closer to users for faster performance.

  • Origins — S3 bucket, ALB, EC2, custom HTTP server
  • Caching — Cache content at edge for defined TTL (Time-to-Live)
  • HTTPS — SSL/TLS termination at the edge
  • OAC (Origin Access Control) — Restrict S3 to only accept requests from CloudFront
  • Lambda@Edge — Run serverless code at edge locations
💡 ANALOGY

CloudFront is like having copies of your store's products stocked in convenience stores worldwide. A customer in Tokyo gets the product instantly from the local store — not shipped from your US warehouse each time.

🔍
Route 53 — DNS Service
▶

Route 53 is AWS's highly available and scalable DNS (Domain Name System) service. It translates domain names (myapp.com) to IP addresses.

Routing Policies

Observability
Monitoring & Logging
CloudWatch, CloudTrail, and AWS Config for full system visibility.
📊
📊
Amazon CloudWatch
▶

CloudWatch is AWS's monitoring and observability service. Think of it as the health dashboard for your entire AWS environment.

📈 Metrics

Collect and track metrics (CPU, memory, disk I/O, network). Create custom metrics from your app.

🚨 Alarms

Trigger actions when metrics exceed thresholds. Send SNS alert, scale ASG, or stop an EC2 instance.

📝 Logs

Centralized log management. Collect logs from EC2, Lambda, RDS, VPC Flow Logs, CloudTrail.

📋 Dashboards

Create custom visual dashboards with graphs, numbers, and alarms across multiple services.

🕵️
AWS CloudTrail — API Audit Logging
▶

CloudTrail records every API call made in your AWS account — who did what, when, and from where. Essential for security auditing and compliance.

  • Logs are stored in S3 and optionally in CloudWatch Logs
  • Records: user identity, action performed, resources affected, timestamp, source IP
  • Enabled by default (90-day history in console). Create a "Trail" for unlimited history.
  • Use case: "Who deleted that S3 bucket?" → CloudTrail has the answer
💡
CloudWatch vs CloudTrail: CloudWatch = WHAT is happening (performance metrics). CloudTrail = WHO did WHAT (API audit log).
⚙️

AWS Config

Continuously audits and records your AWS resource configurations. Checks for compliance against rules (e.g., "all S3 buckets must have encryption"). Tracks configuration history.

Trusted Advisor

AWS's automated best-practice checker. Reviews your account for cost optimization, security, fault tolerance, performance, and service limits. Think of it as an automated consultant.

AWS CloudFormation
▶

CloudFormation lets you provision all your AWS infrastructure in YAML or JSON templates. Define your architecture in code and deploy it consistently, repeatedly.

  • Creates a Stack of resources from a template
  • If stack creation fails, it automatically rolls back
  • Version-controlled infrastructure (store templates in Git)
  • Free! You only pay for the resources it creates
CloudFormation Template (YAML) — S3 Bucket
AWSTemplateFormatVersion: '2010-09-09'
Description: 'Simple S3 Bucket Stack'

Resources:
  MyS3Bucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: my-unique-bucket-name-2024
      VersioningConfiguration:
        Status: Enabled

Outputs:
  BucketName:
    Value: !Ref MyS3Bucket
🌱
Elastic Beanstalk
▶

Elastic Beanstalk is a PaaS service. You upload your code and AWS automatically handles deployment, load balancing, auto-scaling, and monitoring.

  • Supports: Node.js, Python, Ruby, Java, .NET, PHP, Go, Docker
  • You retain full control of the underlying EC2 resources
  • Free — pay only for the AWS resources (EC2, RDS, etc.)
  • Best for developers who want to focus on code, not infrastructure
💡 ANALOGY

Elastic Beanstalk is like Heroku for AWS. Drop in your code, and AWS sets up everything automatically — the servers, load balancer, deployment pipeline.

Messaging: SQS & SNS
Decouple your application components for resilience and scalability.
📨
📬
Amazon SQS — Simple Queue Service
▶

SQS is a managed message queue that lets you decouple and scale distributed applications. Producers send messages; consumers process them at their own pace.

💡 ANALOGY

SQS is like a ticket queue at a concert. People line up (messages added to queue), and staff (consumers) serve one person at a time. No one is turned away, and service happens at a sustainable pace.

Standard Queue

Unlimited throughput. Best-effort ordering. At-least-once delivery. Messages may arrive out of order or duplicate.

FIFO Queue

Exactly-once processing. First-In-First-Out order guaranteed. Up to 300 msg/sec (or 3,000 with batching).

📢
Amazon SNS — Simple Notification Service
▶

SNS is a pub/sub messaging service. A publisher sends one message to a Topic, and it's instantly delivered to all subscribers.

  • Subscribers can be: SQS queues, Lambda functions, HTTP endpoints, email, SMS, mobile push
  • Fan-out pattern: one SNS message → multiple SQS queues simultaneously
  • Real-time delivery (vs SQS which is polled)
💡 ANALOGY

SNS is like a radio broadcast (publish). When you speak (publish a topic), everyone tuned in (subscribers) hears it simultaneously. SQS is like a voicemail (queue) — messages wait until picked up.

What are Containers & Docker?
▶

Containers package your application code + dependencies into a portable, lightweight unit that runs consistently everywhere.

💡 ANALOGY

A container is like a shipping container. It holds everything the app needs. Whether loaded onto a ship (AWS), truck (Azure), or train (GCP), it works the same way — no surprises at delivery.

ECS — Elastic Container Service
▶

ECS is AWS's native container orchestration service. It manages running, stopping, and scaling Docker containers.

ECS on EC2

You manage the underlying EC2 instances (container hosts). More control over infrastructure.

ECS on Fargate ✨

Serverless containers — AWS manages the host. You only define CPU/memory and your container. Zero server management.

☸️
EKS — Elastic Kubernetes Service
▶

EKS is a managed Kubernetes (K8s) service on AWS. If your team already uses Kubernetes or needs maximum portability/power, choose EKS.

  • AWS manages the Kubernetes control plane (API server, etcd)
  • You manage worker nodes (or use Fargate for serverless nodes)
  • Best for complex microservices, multi-cluster setups, teams with K8s expertise

🤝
Shared Responsibility Model
▶

AWS and the customer share security responsibilities — but in different areas.

🏢 AWS Responsible For

"Security of the cloud" — Physical data center security, hardware, hypervisor, managed service software (RDS patching, etc.)

👤 Customer Responsible For

"Security in the cloud" — Your OS patching, application security, data encryption, IAM permissions, network config

🔑
KMS, Secrets Manager & ACM
▶

🔑 KMS

AWS Key Management Service. Create and manage encryption keys for data at rest. Used by S3, RDS, EBS, and most AWS services.

🔒 Secrets Manager

Store and rotate database passwords, API keys, and other secrets. Applications retrieve secrets via API — no hardcoded credentials.

📜 ACM

AWS Certificate Manager. Free SSL/TLS certificates for your domains, integrated with ALB and CloudFront.

🛡️
AWS Shield & WAF
▶

🛡️ AWS Shield

Standard (free): Automatic DDoS protection for all AWS customers. Protects against Layer 3/4 attacks.
Advanced ($3000/month): Enhanced protection, real-time visibility, cost protection.

🧱 AWS WAF

Web Application Firewall. Filter HTTP requests based on rules (SQL injection, XSS, geo-blocking, rate limiting). Integrates with ALB, CloudFront, API Gateway.
