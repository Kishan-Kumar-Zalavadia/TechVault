01

The Problem React Solves

Before React, building interactive UIs meant manually finding DOM elements with getElementById, updating them, keeping them in sync with data, and hoping nothing broke. As apps got bigger, this became a nightmare. React was Facebook's solution in 2013: describe what your UI should look like for any given state, and let React figure out how to efficiently update the DOM.

// THE CORE REACT MENTAL MODEL
UI = f(state)

Your UI is a pure function of your state. Change the state → React recomputes the UI. You never touch the DOM directly.

React vs Vanilla JS

counter.html
// Vanilla JS — you manually touch the DOM every time
let count = 0;
const btn = document.getElementById('btn');
const display = document.getElementById('count');

btn.addEventListener('click', () => {
  count++;
  display.textContent = count; // manually update DOM
  // Imagine 50 UI elements you need to keep in sync...
});
Counter.jsx
// React — describe the UI, React handles the DOM
function Counter() {
  const [count, setCount] = useState(0);
  // When state changes, React re-renders automatically
  return <button onClick={() => setCount(c => c + 1)}>
    Count: {count}
  </button>;
}

JSX is Syntactic Sugar

JSX looks like HTML but it's really JavaScript. Babel compiles every JSX expression to a React.createElement() call. You never need to know this to use React, but it explains why JSX has its quirky rules.

Complete JSX Rules

1️⃣
Single Root Element
▶

Every JSX expression must return ONE root element. Use <>...</> (Fragment) to wrap without adding a DOM node.

Fragment usage
// ❌ Two root elements — error!
return <h1>Title</h1> <p>Text</p>;

// ✅ Wrapped in Fragment
return (
  <>
    <h1>Title</h1>
    <p>Text</p>
  </>
);
2️⃣
CamelCase Attributes
▶

HTML attributes become camelCase. Special cases: class→className, for→htmlFor.

HTML vs JSX
// HTML:   onclick,   tabindex,   maxlength
// JSX:    onClick,   tabIndex,   maxLength

<input
  onClick={handleClick}
  tabIndex={0}
  maxLength={100}
/>
3️⃣
Curly Braces { } for JS
▶

Any JavaScript expression goes inside {}. You can use variables, ternaries, function calls — but NOT statements like if or for.

Expressions in JSX
{name}                    // variable
{2 + 2}                    // arithmetic
{isLoading ? '...' : data}  // ternary
{items.map(i => ...)}       // array map
{formatDate(date)}          // function call
{{color: 'red'}}            // inline styles (double {{}})
4️⃣
Spread Props
▶

Use spread to pass all properties of an object as props. Useful for forwarding props to child components.

Prop spreading
const props = { href: '/home', className: 'link' };
<a {...props}>Home</a>
// Same as: <a href="/home" className="link">Home</a>

Function Components

A function component is just a JavaScript function that accepts props and returns JSX. React calls it during rendering.

UserCard.jsx
// 1. Function name MUST start with capital letter (React uses this to distinguish
//    your components from HTML tags like <div>, <button>)
function UserCard({ name, role, avatar, isOnline }) {
  // 2. Can contain any JavaScript logic
  const initials = name.split(' ').map(n => n[0]).join('');

  // 3. Must return JSX (or null to render nothing)
  return (
    <div className="card">
      <div className="avatar">
        {avatar ? <img src={avatar} alt={name} /> : <span>{initials}</span>}
        <div className={`status ${isOnline ? 'online' : 'offline'}`} />
      </div>
      <h3>{name}</h3>
      <p>{role}</p>
    </div>
  );
}

// Usage:
<UserCard name="Kishan Kumar" role="Full-Stack Engineer" isOnline />

Class Components (Legacy)

⚠️Class components are legacy — you'll see them in older codebases. New code should always use function components + hooks. Here's what they look like so you can read them:
OldCounter.jsx
class OldCounter extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
  }

  render() {
    return (
      <button onClick={() => this.setState({ count: this.state.count + 1 })}>
        Count: {this.state.count}
      </button>
    );
  }
}

// Modern equivalent with hooks (simpler, cleaner):
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>;
}

Composition over Inheritance

React favors composition — building complex UIs by nesting simpler components — over class inheritance. The children prop is your key to creating flexible wrapper components.

Composition pattern
// A generic wrapper that can hold ANY children
function Card({ title, footer, children }) {
  return (
    <div className="card">
      {title && <div className="card-header">{title}</div>}
      <div className="card-body">{children}</div>
      {footer && <div className="card-footer">{footer}</div>}
    </div>
  );
}

// Compose freely:
<Card title="User Profile" footer={<Button>Edit</Button>}>
  <Avatar user={user} />
  <Bio text={user.bio} />
</Card>

useState — Every Pattern

Primitive state
const [count, setCount] = useState(0);
const [name, setName] = useState('');
const [isOpen, setIsOpen] = useState(false);

// Updating:
setCount(5);              // set directly
setName('Kishan');        // set directly
setIsOpen(prev => !prev); // toggle
Object state — always spread!
const [user, setUser] = useState({
  name: '', email: '', age: 0, isAdmin: false
});

// ❌ WRONG — replaces the entire object!
setUser({ name: 'Kishan' }); // email, age, isAdmin are LOST

// ✅ CORRECT — spread existing, then override one field
setUser(prev => ({ ...prev, name: 'Kishan' }));

// Deeply nested update (use immer for complex cases)
setUser(prev => ({
  ...prev,
  address: { ...prev.address, city: 'Dallas' }
}));
Array state — never mutate!
const [items, setItems] = useState([]);

// ADD — spread existing + new item
setItems(prev => [...prev, newItem]);

// REMOVE — filter out the item
setItems(prev => prev.filter(item => item.id !== targetId));

// UPDATE — map and replace matching item
setItems(prev => prev.map(item =>
  item.id === targetId ? { ...item, done: true } : item
));

// INSERT AT INDEX
setItems(prev => [
  ...prev.slice(0, index),
  newItem,
  ...prev.slice(index)
]);

// ❌ NEVER DO THESE — mutations don't trigger re-render:
items.push(newItem);     // ❌
items[0].done = true;   // ❌
items.splice(0, 1);     // ❌
Lazy initialization — for expensive initial state
// ❌ This runs on EVERY render (expensive!)
const [data, setData] = useState(parseHeavyData(rawData));

// ✅ Pass a function — only runs ONCE on mount
const [data, setData] = useState(() => parseHeavyData(rawData));

// Common use case: reading from localStorage
const [theme, setTheme] = useState(() => {
  return localStorage.getItem('theme') ?? 'dark';
});
Functional update — when new state depends on old
// PROBLEM: stale state in closures
function handleClick() {
  setCount(count + 1); // count captured at render time
  setCount(count + 1); // still the same old count! only +1
}

// SOLUTION: pass a function — React provides latest state
function handleClick() {
  setCount(c => c + 1); // always fresh
  setCount(c => c + 1); // queued after first — result: +2 ✅
}

// Rule: if new value depends on old → use functional form
🔴Golden Rule: Never mutate state directly. React compares state by reference (===). If the reference doesn't change, React thinks nothing changed and skips re-rendering — your UI goes stale.

07

Synthetic Events

React wraps native browser events in a SyntheticEvent — a cross-browser wrapper that normalizes behavior. The API is identical to native events (event.target, event.preventDefault(), etc.).

EventPatterns.jsx
// ── INLINE ARROW FUNCTION ──
<button onClick={() => setCount(c => c + 1)}>+</button>

// ── NAMED HANDLER (preferred for complex logic) ──
function handleClick(event) {
  event.preventDefault();   // stop default browser action
  event.stopPropagation(); // stop bubbling to parent
  setCount(c => c + 1);
}
<button onClick={handleClick}>+</button>

// ── PASSING DATA TO HANDLER ──
const handleDelete = (id) => () => deleteItem(id); // curried
{items.map(item => (
  <button onClick={handleDelete(item.id)}>Delete</button>
))}

// ── COMMON EVENTS ──
<input
  onChange={e => setValue(e.target.value)}   // text input
  onFocus={() => setFocused(true)}
  onBlur={() => setFocused(false)}
  onKeyDown={e => e.key === 'Enter' && submit()}
/>

<form onSubmit={e => { e.preventDefault(); handleSubmit(); }}>

<div
  onMouseEnter={() => setHovered(true)}
  onMouseLeave={() => setHovered(false)}
>
💡Event Delegation: React attaches ONE event listener at the root (not on each element). This is why React events are efficient even with thousands of buttons in a list.

TodoList.jsx
const todos = [
  { id: 'a1b2', text: 'Learn React hooks', done: true },
  { id: 'c3d4', text: 'Build a project', done: false },

ConditionalPatterns.jsx
// ── PATTERN 1: Early Return (cleanest for big branches) ──
function Profile({ user, isLoading, error }) {
  if (isLoading) return <Skeleton />;
  if (error)     return <Error message={error}/>;
  if (!user)     return null;  // render nothing
  return <UserCard {...user} />;
}

// ── PATTERN 2: Ternary (for two clear options) ──
{isLoggedIn ? <Dashboard /> : <LoginPage />}

// ── PATTERN 3: && short-circuit (show or nothing) ──
{hasNotifications && <Badge count={count} />}

// ── PATTERN 4: ?? nullish coalescing ──
{user.bio ?? 'No bio provided'}

// ── GOTCHA: 0 && <Thing/> renders 0! ──
// items.length && <List/>  ← if length is 0, renders "0" on screen!
// Fix: use Boolean() or explicit check:
{items.length > 0 && <List />}
{!!items.length && <List />}

// ── PATTERN 5: Object map (replacing long if-else chains) ──
const statusComponents = {
  loading: Spinner,
  error:   ErrorView,
  empty:   EmptyState,
  success: DataView,
};
const Component = statusComponents[status];
return <Component {...props} />;

useEffect — The Complete Guide
Side effects, the dependency array, cleanup functions, and all the subtle gotchas that trip up every React developer.
11
⏱️ Component Lifecycle Timeline
// Click a lifecycle phase to see what runs
🟦 1. Component function runs
React calls your function, runs all the code top to bottom, builds JSX.
🟩 2. DOM painted (browser renders)
React commits the Virtual DOM diff to the real DOM. The user can see the UI.
🟨 3. useEffect runs (after paint)
Effects run asynchronously AFTER the browser has painted. Good for data fetching, subscriptions.
🟪 4. State/props change → re-render
Steps 1-3 repeat. Before step 3, the cleanup from the previous effect runs first.
🟥 5. Cleanup runs on unmount
When the component leaves the tree, React runs your cleanup function one final time.

Dependency Array — The Core Concept

useEffect patterns
// ── CASE 1: No dependency array → runs after EVERY render ──
useEffect(() => {
  document.title = `Count: ${count}`; // fine, but runs too often
}); // ← no array

// ── CASE 2: Empty array [] → runs ONCE on mount ──
useEffect(() => {
  const ws = new WebSocket('wss://api.example.com');
  return () => ws.close(); // cleanup on unmount
}, []); // ← empty array = mount only

// ── CASE 3: Specific deps → runs when they change ──
useEffect(() => {
  document.title = `Page: ${pageId}`;
}, [pageId]); // only runs when pageId changes

// ── CASE 4: Data fetching with cleanup ──
useEffect(() => {
  const controller = new AbortController();
  async function fetchData() {
    try {
      const res = await fetch(`/api/posts/${id}`, { signal: controller.signal });
      const data = await res.json();
      setPosts(data);
    } catch (err) {

Common useEffect Mistakes

❌
Missing dependencies
▶

If you use a value inside useEffect but don't list it in dependencies, your effect uses a stale (old) version of that value. The ESLint rule exhaustive-deps catches this automatically.

Stale closure
// ❌ user not in deps — uses stale user on every call
useEffect(() => {
  sendWelcome(user); // user could be outdated!
}, []); // should be [user]
❌
Infinite loops
▶

Setting state inside an effect that depends on that state creates an infinite loop: state changes → effect runs → sets state → effect runs → …

Infinite loop
// ❌ LOOP: data changes → effect → setData → data changes → ...
useEffect(() => {
  setData(processData(data)); // data is in deps below!
}, [data]);

// ✅ FIX: use useMemo instead for derived state
const processed = useMemo(() => processData(data), [data]);
❌
No cleanup for subscriptions
▶

Without cleanup, event listeners and subscriptions pile up every render, causing memory leaks and duplicate events.

Memory leak
// ❌ New listener added on every render, old ones never removed
useEffect(() => {
  window.addEventListener('resize', handleResize);
}); // no array + no cleanup = memory leak

// ✅ Cleanup removes the listener
useEffect(() => {
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize);
}, []);
❌
Async directly in useEffect
▶

useEffect's callback can't be async because an async function returns a Promise, but useEffect expects either nothing or a cleanup function.

Fix: inner async function
// ❌ Can't do this:
useEffect(async () => { // returns Promise — React ignores it
  const data = await fetchData();
}, []);

// ✅ Define async inside, then call it:
useEffect(() => {
  async function load() {
    const data = await fetchData();
    setData(data);
  }
  load();
}, []);

function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM': {
      const existing = state.items.find(i => i.id === action.item.id);
      if (existing) {
        return {
          ...state,
          items: state.items.map(i =>
            i.id === action.item.id ? { ...i, qty: i.qty + 1 } : i
          ),
          total: state.total + action.item.price,
        };
      }
      return {
        ...state,
        items: [...state.items, { ...action.item, qty: 1 }],
        total: state.total + action.item.price,
      };
    }
    case 'REMOVE_ITEM': {
      const item = state.items.find(i => i.id === action.id);
      return {
        ...state,
        items: state.items.filter(i => i.id !== action.id),
        total: state.total - (item?.price * item?.qty ?? 0),
      };
    }
    case 'APPLY_DISCOUNT':
      return { ...state, discount: action.percent };
    case 'CLEAR':
      return initialState;
    default:
      return state;
  }
}

useLayoutEffect vs useEffect
Understand the subtle but critical timing difference — and when each is appropriate.
17

useEffect

  • Runs after the browser paints
  • Async — doesn't block rendering
  • ✅ Data fetching, subscriptions, timers
  • ✅ Almost everything you need
  • User sees the painted UI before effect runs

useLayoutEffect

  • Runs synchronously after DOM mutations, before paint
  • Blocks browser paint until done
  • ✅ Reading DOM measurements (width, height, position)
  • ✅ Preventing flashes of wrong layout
  • ⚠️ Use sparingly — can slow rendering
When to use useLayoutEffect
// PROBLEM: Tooltip position flashes to wrong position then corrects

  const rect = triggerRef.current.getBoundingClientRect();

// SOLUTION: useLayoutEffect fires BEFORE paint

  const rect = triggerRef.current.getBoundingClientRect();
  setTooltipPos({ top: rect.bottom, left: rect.left });
}, []);

// RULE: Default to useEffect. Only switch to useLayoutEffect
// when you see a visual flicker caused by DOM measurements.

function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);
    fetch(url)
      .then(r => r.json())
      .then(d => { if (!cancelled) { setData(d); setLoading(false); } })
      .catch(e => { if (!cancelled) { setError(e); setLoading(false); } });
    return () => { cancelled = true; };
  }, [url]);

  return { data, loading, error };
}

⚠️Without error boundaries: a JS error in any component crashes your entire React app — blank screen. Error boundaries are your safety net.
Error Boundary class component
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, { componentStack }) {
    logToSentry(error, componentStack);
  }

  render() {
    if (this.state.hasError) {
      return (
        <div>
          <h2>Something went wrong 😔</h2>
          <button onClick={() => this.setState({ hasError: false })}>Try again</button>
        </div>
      );
    }
    return this.props.children;
  }
}

🕵️ Re-render Detective — Spot the Problem

Case 1 — Why does the memoized child still re-render?
function Parent() {
  return <Child style={{ color: 'red' }} />;  ← ?
}
❌ New object on every render. {{ color: 'red' }} creates a new reference each render — React.memo sees a changed prop.
✅ Fix: const style = useMemo(() => ({ color: 'red' }), [])
Case 2 — Inline callback defeats memo
<ExpensiveList onItemClick={item => setSelected(item)} />  ← ?
❌ New function every render. Even with React.memo, the callback prop changes reference each time.
✅ Fix: const handleClick = useCallback(item => setSelected(item), [])
Case 3 — State too high in the tree
function App() {
  const [tooltip, setTooltip] = useState('');  ← ?
  return <><Header /><MainContent /><Footer tooltip={tooltip}/></>;

✅ Fix: Move tooltip state into Footer — colocate state close to where it's used.

⚠️React Native has a JS thread and a UI thread. Heavy JS work can cause dropped frames. Move animations and gestures to the UI thread with Reanimated.
Performance tips
// ✅ Use react-native-reanimated for smooth animations
import Animated, { useSharedValue, withSpring } from 'react-native-reanimated';

function AnimatedCard() {
  const scale = useSharedValue(1);  // runs on UI thread
  return (
    <Animated.View style={{ transform: [{ scale }] }}>
      <TouchableOpacity
        onPressIn={() => { scale.value = withSpring(0.95); }}
        onPressOut={() => { scale.value = withSpring(1); }}
      >
        <Text>Press me!</Text>
      </TouchableOpacity>
    </Animated.View>
  );
}
